# Feature Architecture Follow-up Plan

## Completion rule

Each phase is complete only when the affected runtime path follows the manifesto architecture, its contribution is
discoverable, behavioral validation exercises the relationship, production composition binds it exactly once, and
application consumers no longer retain an alternate manual path.

Compilation is an outcome of completing a phase, not a reason to preserve curated dispatch or duplicated authorities.

## Phase 1: Durable participation architecture

Extend executable Feature participation with owner-contributed durable preparation, stable payload ownership, and
durable delivery. Generic workflow hosts may persist opaque participant envelopes and delivery state, but must not know
participant IDs, payload schemas, or handlers.

Validation must reject missing, duplicate, orphaned, unbound, or uncontracted durable participants. Unknown persisted
participants must have an explicit compatibility policy rather than falling into a generic coordinator switch.

Milestone: the architecture can prove an unknown test participant enters durable preparation and delivery without a
curated edit.

## Phase 2: Migration durable consequences

Migrate Progress, Playback Preferences, Viewer Settings, Download removal, Tracking transfer where applicable, and the
custom-cover host consequence to independently owned durable participants. Remove Migration's artifact switch and fixed
Feature dependency list.

Milestone: a new durable participant can join Migration solely through its contribution and runtime binding.

## Phase 3: Merge durable consequences

Migrate Tracking initialization, Download removal, and custom-cover cleanup to durable participants. Remove Merge's
artifact switch and fixed consequence construction.

Milestone: Merge persists and retries discovered opaque consequences without naming their owners.

## Phase 4: Media-session lifecycle

Introduce the structured session event boundary and separate event facts from Feature consequences. Migrate Manga,
Anime, Book, and immersive runtimes to emit events. Contribute Progress persistence, History recording, Tracking
synchronization, Download lifecycle behavior, and incognito policy through owned participation.

Milestone: every media runtime uses the same lifecycle seam, current behavior is preserved intentionally, and adding a
new media implementation exposes missing specialized obligations rather than silently omitting consequences.

## Phase 5: Refresh lifecycle

Add execution points for newly discovered children in manual Entry refresh and Library update. Migrate Automatic
Download to owned participants while preserving deferred batch start for Library updates. Keep worker scheduling,
progress notifications, and failure reporting in the app workflow.

Milestone: refresh coordinators no longer call Automatic Download directly and a test participant joins both applicable
refresh contexts through contribution.

## Phase 6: Lifecycle and consumer corrections

Route permanent Profile deletion through discovered Entry destructive-removal and any required profile-wide state
participants. Route Manga Reader bookmarking through `EntryBookmarkFeature`. Add boundary checks preventing equivalent
direct persistence paths from returning.

Milestone: cleanup and Bookmark consequences cannot be bypassed by the corrected consumers.

## Phase 7: Settings projections

Generalize app-owned settings projection so one contributed destination drives main navigation, search indexing, and
visibility. Migrate Feature-owned settings surfaces without turning generic settings into Entry Features.

Milestone: production validation rejects an applicable Feature settings projection that is missing, duplicated, or not
reachable from both navigation and search.

## Phase 8: Final enforcement and review

Run the complete architecture gate, focused tests, FOSS compilation, telemetry-enabled release compilation, formatting,
and documentation verification. Re-run the architecture-wide audit probes. Review deferred candidates separately and
do not absorb them without an explicit decision.

Milestone: no approved gap or parallel authority remains, generated reporting reflects all new relationships, and the
working tree contains no implementation-only planning artifacts that are no longer useful.
